# 2026-03-14 — Qwen3.5-35B-A3B on vLLM PP=3 (3× RTX 3060)

## Goal
Validate whether vLLM pipeline parallelism can deliver higher aggregate throughput and fair multi-request scheduling on a non-NVLink 3× RTX 3060 setup, while keeping 131K context support.

---

## Setup

### Hardware
- 3× RTX 3060 12GB (single host, mixed PCIe lanes)
- 24GB RAM LXC runtime for vLLM

### Software
- vLLM 0.17.1
- Model: `Qwen/Qwen3.5-35B-A3B-GPTQ-Int4` (official)
- Quant backend: `gptq_marlin`

### Final serving profile (stable + fast)
```bash
vllm serve /mnt/models/gptq/Qwen3.5-35B-A3B-GPTQ-Int4 \
  --host 0.0.0.0 --port 8000 \
  --pipeline-parallel-size 3 \
  --kv-cache-dtype fp8_e4m3 \
  --max-model-len 131072 \
  --max-num-seqs 3 \
  --max-num-batched-tokens 8192 \
  --language-model-only \
  --performance-mode interactivity \
  --mamba-cache-mode align \
  --mamba-block-size 8 \
  --enable-prefix-caching \
  --enable-chunked-prefill \
  --reasoning-parser qwen3 \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder \
  --gpu-memory-utilization 0.88 \
  --compilation-config '{"cudagraph_capture_sizes": [1,2,3]}'
```

### Environment variables
```bash
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True,max_split_size_mb:512
VLLM_USE_FLASHINFER_MOE_FP16=1
VLLM_SLEEP_WHEN_IDLE=1
VLLM_PP_LAYER_PARTITION=14,13,13
OMP_NUM_THREADS=4
CUDA_HOME=/usr/local/cuda
```

---

## Commands (high level)
1. Enable 3rd GPU passthrough to vLLM runtime.
2. Upgrade vLLM to 0.17.1.
3. Install missing FlashInfer JIT dependencies:
   - `cuda-nvcc-12-5`
   - `ninja-build`
4. Download official GPTQ model.
5. Tune launch profile from:
   - eager mode (`--enforce-eager`, stable but ~19 tok/s)
   - to CUDAGraph mode (fast), then reduce graph memory overhead via `cudagraph_capture_sizes=[1,2,3]`.
6. Run single-request and concurrency benchmarks with `vllm bench serve`.

---

## Observations

### Dependency gotchas
- FlashInfer JIT requires both `nvcc` and `ninja`.
- Without them, first inference fails even if server starts.

### Performance mode tradeoff
- Eager mode was stable but slower (~19 tok/s).
- CUDAGraph mode reached ~60 tok/s after warmup, but default captures consumed too much VRAM.
- Restricting capture sizes to `[1,2,3]` restored a usable KV cache without losing steady-state TG speed.

### Warmup behavior
- First 1–2 requests can be very slow due to JIT + graph capture.
- After warmup, token generation stabilizes near 58–62 tok/s.

### Architecture behavior
- This model uses hybrid DeltaNet + attention blocks.
- vLLM reports KV in tokens for the attention cache pool; fixed-state DeltaNet memory is separate.

## Metrics

### Server startup metrics
- Layer split: `[14, 13, 13]`
- Available KV cache memory: **1.91 GiB**
- GPU KV cache size: **88,032 tokens**
- Reported max concurrency @ 131,072 context: **2.48×**

### Single-request TG (post-warmup)
- 183 tok in 3.14s → **58.3 tok/s**
- 196 tok in 3.26s → **60.1 tok/s**
- 571 tok in 9.25s → **61.7 tok/s**
- 150 tok in 2.53s → **59.3 tok/s**

### Concurrency benchmark (moderate)
`random_input_len=8192`, `random_output_len=128`, `num_prompts=20`, `request_rate=2`
- Peak concurrent requests: **20**
- Output token throughput: **34.15 tok/s**
- Total token throughput: **2219.84 tok/s**
- TPOT mean: **53.13 ms**

### Concurrency benchmark (stress)
`random_input_len=8192`, `random_output_len=256`, `num_prompts=50`, `request_rate=10`
- Peak concurrent requests: **50**
- Output token throughput: **54.26 tok/s**
- Peak output throughput: **130 tok/s**
- Total token throughput: **1790.52 tok/s**
- TPOT mean: **39.34 ms**
- ITL mean: **39.26 ms**

---

## Conclusion
- vLLM PP=3 is viable and significantly improves aggregate multi-request throughput on 3× 3060.
- With tuned CUDAGraph capture sizes, setup reaches ~60 tok/s steady TG while preserving substantial KV headroom.
- This profile is a strong candidate for high-concurrency agent workloads.
