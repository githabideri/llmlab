# Qwen3.6-35B-A3B-MTP on Single RTX 3060 12GB

**Date:** 2026-06-30
**Category:** Model deployment
**Hardware:** RTX 3060 12GB (compute cap 8.6), i3-9100 4C/4T, 16 GB RAM
**Stack:** llama.cpp b9850 (4f31eed), CUDA 13.1, GGML_CUDA

---

## Summary

Qwen3.6-35B-A3B-MTP (UD-Q4_K_XL, 22 GB) runs on a single RTX 3060 12GB using hybrid CPU+GPU offload. The key finding: `--n-cpu-moe 26` is the sweet spot, delivering **33-36 t/s completion** and **50+ t/s prompt processing**. The research-doc default of `--n-cpu-moe 32` was half as fast (16.5 t/s).

| Metric | Value |
|--------|-------|
| Model | Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf (22 GB) |
| VRAM used | 9.9 / 12.3 GB (80%) |
| Completion | 33-36 t/s |
| Prompt | 50-58 t/s |
| MTP acceptance | 91% (31/34 drafts) |
| Context | 4096 (larger OOMs on 16 GB RAM) |

## How MoE Offload Works

MoE models have two weight classes:
- **Dense** (~2.5 GB for this model): attention, router, shared FFN — used for every token
- **Experts** (~19.5 GB): routed FFN — only a subset activated per token per layer

llama.cpp's offload flags control where each class lives:

| Flag | Effect |
|------|--------|
| `-ngl 99` | Offload all layers to GPU (model has 41 layers, so 99 = "all") |
| `--n-cpu-moe N` | Keep expert weights for the **first N layers** on CPU instead of GPU |

With `-ngl 99 --n-cpu-moe 26`:
- Dense layers (all 41) → GPU
- Experts for layers 0-25 → CPU RAM
- Experts for layers 26-40 → GPU VRAM
- Total VRAM: ~9.9 GB

Sources: [llama.cpp source](https://github.com/ggml-org/llama.cpp/blob/master/common/arg.cpp) (`--n-cpu-moe` iterates `for (int i = 0; i < value; ++i)`), [DEV Community explanation](https://dev.to/someoddcodeguy/understanding-moe-offloading-5co6), [DocShotgun MoE guide](https://gist.github.com/DocShotgun/a02a4c0c0a57e43ff4f038b46ca66ae0)

## n-cpu-moe Sweep

Completion throughput varies non-linearly with `--n-cpu-moe`. Anomalies at 24 and 28 suggest layer-boundary effects (model has 41 layers).

| --n-cpu-moe | VRAM | Prompt t/s | Completion t/s |
|
|-----------|------|------------|----------------|
| 32 | 8.1 GB | 32.8 | 16.5 |
| 30 | 8.0 GB | 36.4 | 24.8 |
| 29 | 8.5 GB | 39.8 | 30.2 |
| 28 | 8.9 GB | 41.2 | 10.4 ❌ |
| 27 | 9.4 GB | 41.6 | 31.8 |
| **26** | **9.9 GB** | **37.9** | **33.6** ✅ |
| 25 | 10.3 GB | 36.2 | 32.6 |
| 24 | 10.8 GB | 11.2 | 17.9 ❌ |
| 23 | 11.2 GB | 14.9 | 14.9 ❌ |
| 20 | OOM | — | — |

**Sweet spot: `--n-cpu-moe 26`** (33.6 t/s completion, 9.9 GB VRAM).

## Working Config

```bash
llama-server \
  -m /path/to/Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf \
  --spec-type draft-mtp --spec-draft-n-max 2 \
  -ngl 99 --n-cpu-moe 26 \
  -c 4096 \
  -ctk q8_0 -ctv q8_0 \
  --flash-attn on \
  --temp 0.6 --top-p 0.95 --top-k 20 \
  --no-mmap \
  -np 1 \
  --jinja \
  --host 0.0.0.0 --port 8080
```

**Build flags:**
```bash
cmake -B build \
  -DGGML_CUDA=ON \
  -DCMAKE_CUDA_ARCHITECTURES=86 \
  -DGGML_CUDA_GRAPHS=ON \
  -DGGML_CUDA_F16=ON \
  -DGGML_CUDA_FA_ALL_QUANTS=ON
```

## 128K Context Benchmarks (2026-07-02, Gen3 PCIe, 20 GB RAM)

After fixing PCIe Gen1→Gen3 (was power-saving idle state, jumps to Gen3 under load) and increasing LXC RAM to 20 GB, full 128K context works.

**Best config:** `--n-cpu-moe 28 -ub 1536 -c 131072 -ctk q8_0 -ctv q4_0`

### Degradation Curve

| Filled Context | Prompt t/s | Complete t/s | MTP % | Prompt time | Complete time |
|----------------|-----------|-------------|-------|-------------|---------------|
| 1,110 tok | 521.7 | 38.7 | 84.9% | 2.1s | 13.0s (500 tok) |
| 67,976 tok | 643.0 | 33.1 | 83.2% | 105.7s | 6.0s (200 tok) |
| 126,166 tok | 562.3 | 28.8 | 82.6% | 224.3s | 6.9s (200 tok) |

**vs original (Gen1, 4K ctx):** Prompt 37.9 → 521-643 t/s (14-17×). Completion 33.6 → 28-39 t/s (stable).

### Key Findings

- **Prompt speed is PCIe-bandwidth bound, not KV-cache bound.** At long context, larger batches amortize PCIe transfer cost → throughput stays high (560-643 t/s).
- **Completion degrades 15-26% at long context** from q4_0 V-cache dequant overhead. Matches DGX Spark benchmark expectations.
- **MTP acceptance stable** (82-85%) across all context lengths.
- **`-ub` matters more than `--n-cpu-moe`** for prompt speed. `-ub 1536` beats `-ub 512` by 57% at same `--n-cpu-moe`.
- **Full 128K prefill: ~3.7 minutes.** The main bottleneck for real-world use.

### VRAM Budget at 128K

| --n-cpu-moe | -ub | VRAM | Fits? |
|-------------|-----|------|-------|
| 26 | 512 | 11.6 GB | ✅ |
| 26 | 1024 | OOM | ❌ |
| 28 | 1024 | 11.0 GB | ✅ |
| 28 | 1536 | 11.35 GB | ✅ |
| 28 | 2048 | OOM | ❌ |

### Remaining Optimizations

1. **Faster prefill:** Need more VRAM for higher `-ub`, or multi-GPU tensor-split
2. **Faster completion at long context:** Switch to f16 V-cache (needs ~313 MiB more VRAM at 128K) or wait for TurboQuant merge
3. **`--no-op-offload`:** Not needed at Gen3 — PCIe is fast enough for expert transfers

## Caveats

- **Context limited to 4K** on 16 GB RAM. The model is 22 GB; larger contexts add KV cache pressure that OOMs the container/host.
- **Prompt speed depends on PCIe bandwidth.** Hybrid offload shuttles activations across PCIe for CPU-resident layers. On Gen 3 x16 PCIe, expect 50+ t/s. On Gen 1 x8 (~4 GB/s), prompt processing drops significantly.
- **Layer-boundary anomalies** at `--n-cpu-moe 24` and `28` produce much lower throughput. The model has 41 layers; the sweet spot sits at 26.
- **`--no-mmap` is critical** — without it, the model tries to memory-map the 22 GB file and exceeds `RLIMIT_MEMLOCK`.
- **`--mlock` fails** on large models — `RLIMIT_MEMLOCK` is too small. Drop it from server args.
- **MTP n=2 is the sweet spot** — n=3+ drops acceptance and throughput based on RTX 4070 12GB data [carteakey](https://carteakey.dev/blog/running-qwen3-6-mtp-locally/).
